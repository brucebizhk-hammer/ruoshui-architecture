模块选择、用途、体量与边界

点击上方按钮筛选

Scrapy

直接接入

负责企业站、招投标、行业目录等静态和结构化页面。输出统一 Evidence。

体量：中；Python 库，不新增常驻服务。

避免冲突：不负责动态浏览，不自己写客户库。

风险：低

Playwright

按需接入

只处理必须执行 JavaScript、滚动、点击、登录态的网页。

体量：大；浏览器约数百 MB，任务时启动。

避免冲突：不再同时引入 Crawlee 编排层。

风险：中

MediaCrawler

隔离接入

负责小红书、抖音、微博、B站、贴吧、知乎等中文社媒公开线索。

体量：本地源码约 24.2MB；登录态和反爬维护较重。

避免冲突：独立 Worker，只交付证据，不进入 Core 进程。

风险：高

Trafilatura

直接接入

把网页正文、标题、日期和链接清洗成稳定文本，替代手写正文解析。

体量：中；Python 库，按任务调用。

避免冲突：只做抽取，不做抓取调度和知识判断。

风险：低

RapidFuzz + phonenumbers

直接接入

标准化手机号；辅助识别公司简称、人名别名和重复线索。

体量：轻；库级依赖，无常驻进程。

避免冲突：微信号仍是最高身份键，模糊匹配只给建议。

风险：低

APScheduler

直接接入

负责采集批次、限额节奏、重试和定时补证；状态仍写入数据库。

体量：轻；Python 库，不需要 Redis。

避免冲突：不引入 Celery/Temporal，Core 仍拥有工作流状态。

风险：低

Prometheus Client

直接接入

暴露 Provider 成功率、延迟、成本、重复率、门禁拒绝率和积压。

体量：轻；先只提供 /metrics，不部署完整监控集群。

避免冲突：业务审计仍在 Unified DB。

风险：低

“说人话”规则

裁剪借鉴

提取事实保护、聊天轻改写和 AI 套话检测，形成若水专用风格规则。

体量：仓库约 4.4MB；实际进入提示词仅几 KB。

避免冲突：不是第二个 Agent，不拥有记忆或业务判断。

风险：低

LangGraph

只借鉴

借鉴状态图、暂停、恢复和人工接管模式，不把框架装进现有 Core。

体量：大；完整生态和现有状态机重叠。

冲突：会形成第二套状态、记忆和工作流控制。

风险：高

Firecrawl

首期不采用

抓取和转 Markdown 能力成熟，但与 Scrapy、Playwright、Trafilatura高度重叠。

体量：重；自托管服务链较大，AGPL-3.0。

替代：仅在特殊站点作为付费 API 兜底。

冗余：高

Crawlee

首期不采用

自带队列、存储和浏览器编排，但会与 Scrapy、Playwright、Core 队列重复。

体量：中到大。

替代：需要大规模浏览器池时再重新评估。

冗余：高

Dify

不进入主链

适合低代码 Agent 应用，但会复制 Core 的工作流、模型、知识库和权限。

体量：很重；完整平台和多服务部署。

保留：只参考操作台体验，不作为控制中心。

冲突：很高

Neo4j / Qdrant / Elasticsearch

暂不采用

当前数据量使用 SQLite、FTS5 和关系表足够，暂无独立图数据库和向量库收益。

体量：新增多个常驻服务。

升级条件：知识和关系查询达到明确性能瓶颈。

冗余：高

Wechaty / 新微信框架

禁止并行

现有 DLL 19088 + Middleware 已稳定，新增微信框架会产生双通道、重复回调和风控风险。

体量：不仅是代码，还涉及登录协议和账号风险。

结论：保留现有微信通道。

冲突：最高

最终封装：看起来是一套系统，内部仍可替换

逻辑模块化，物理上不过度微服务化

常驻 1：WeChat Middleware

只负责主号/第二号 I/O、媒体、DLL 和发送回执。不做客户等级、业务判断或模型编排。

常驻 2：Ruoshui Core

单一 API、Policy、Lifecycle、Memory、Action Gateway 和审计。外部工具不能绕开它。

按需 3：Provider Worker

一个 Worker 容纳 Scrapy、Playwright、Trafilatura、MediaCrawler Adapter；每种 Provider 可单独启停。

数据 4：Unified DB + 文件仓

SQLite 保存状态、证据和审计；原始网页、图片、文件进入对象目录。暂不拆多数据库。

旁路 5：Chatwoot / 监控

人工接管、只读展示和指标查看。断开不会影响主链运行。

不允许：第二控制中心

Dify、LangGraph、爬虫框架、Chatwoot、Hermes 都不能同时拥有客户状态、审批和发送权。

最终建议

直接接入 Scrapy、Playwright、Trafilatura、RapidFuzz/phonenumbers、APScheduler 和轻量指标；MediaCrawler 隔离接入；“说人话”只裁剪成若水风格规则。Firecrawl、Crawlee、LangGraph、Dify、独立向量库和新微信框架暂不进入主链。这样能获得成熟能力，同时保持 Core 单一控制权，最终可以封装成一套启动、一套配置、一套审计的一体化系统。
