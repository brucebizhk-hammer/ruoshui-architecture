# 若水客户开发 Agent 流程方案

更新时间：2026-07-07

## 定位

若水不只是被动回复工具，还要承担客户开发职能：

1. 全网持续搜寻客户线索。
2. 对线索做程序初筛，进入预备库。
3. 高相关线索进入待添加库。
4. 按微信风控限量自动加微信。
5. 加微信后默认进入 C 类，自然补档。
6. 符合条件后申请升 B。
7. B 类继续做非敏感业务对接和背景调查。
8. 发现明确成交意向后申请升 A。
9. A 类转为人工主导，若水做辅助分析、汇报和提醒。

## 总流程

```text
人工设定阶段主题/搜索范围
  -> 全网搜索 Agent
  -> Lead 预备库
  -> 程序审核/打分
  -> 待添加库
  -> 限量自动加微信
  -> 默认 C 类
  -> 自然补档
  -> C->B 申请 / C->D 静默
  -> B 类深入对接和背景调查
  -> B->A 申请
  -> A 类人工主导跟进
```

## 1. 全网找客户线索

范围由人工阶段性设定，若水围绕主题展开搜索。

当前主板块：

- 硫磺
- 磷矿
- 石膏
- 能源
- 固废
- 化工贸易

搜索对象：

- 公司
- 个人
- 电话
- 微信号
- 平台主页
- 行业文章
- 招采/供需/物流/贸易信息

线索先进入 `Lead` 预备库，不直接进入客户库。

原因：

- 网上信息模糊、脏、过期、重复。
- 同一个主体可能多个名称。
- 电话、微信、公司名之间关系不稳定。
- 初期不用过度苛刻，先独立沉淀，后续靠规则和人工周期性修正。

## 2. Lead 预备库

建议字段：

| 字段 | 说明 |
| --- | --- |
| `source_platform` | 来源平台 |
| `source_url` | 来源链接 |
| `source_title` | 来源标题 |
| `source_snapshot` | 证据摘要 |
| `person_name` | 姓名 |
| `company_name` | 公司名 |
| `phone` | 电话 |
| `wechat_id` | 微信号 |
| `main_products` | 主营品种 |
| `region` | 地区 |
| `guessed_role` | 角色猜测：资源方、买方、卖方、中间人、物流、资金方、合作伙伴等 |
| `evidence` | 相关证据 |
| `confidence_score` | 信息可信度 |
| `business_relevance_score` | 业务相关度 |
| `risk_flags` | 风险点 |
| `suggested_action` | 建议动作 |
| `review_status` | `reserve` / `promote_to_add_queue` / `discard` |
| `topic_id` | 阶段性搜索主题 |
| `created_at` | 入库时间 |
| `last_seen_at` | 最近发现时间 |

预备库策略：

- 高相关但信息不足：继续补资料。
- 中相关或可疑：留在预备库。
- 达到一定数量或时间后，人工复核一批。
- 人工复核的重点不是逐条审批，而是修正审核程序标准。

## 3. 程序审核与待添加库

用户确认：从预备库到待添加库，不走人工逐条审批，而由审核程序判断。

待添加库标准：

- 高相关。
- 有明确意向或明确业务要点。
- 至少有一种可触达方式：电话或微信号。
- 主营品种或上下游角色与当前主题相关。
- 风险点不高。

预备库保留标准：

- 中相关。
- 信息不完整。
- 主体身份不清。
- 可疑但未达到丢弃标准。
- 等待后续补资料或人工周期性复核。

不入库标准：

- 明显无关。
- 明显诈骗、博彩、资金盘、广告号。
- 无法证明与主题有任何业务关系。
- 只有噪声信息，没有主体、品种、地区或联系方式。

需要支持不同品类、不同等级的审核标准：

- 硫磺线索标准
- 磷矿线索标准
- 石膏线索标准
- 能源线索标准
- 固废线索标准
- 化工贸易线索标准

每套标准都应可调权重。

## 4. 限量自动加微信

用户确认：加微信本身必须自动加，不默认人工介入。

边界：

- 必须来自待添加库。
- 必须通过审核程序。
- 每天按微信风控原则限定人数。
- 需要节奏控制，避免短时间集中添加。
- 需要失败、通过、拒绝、无响应记录。
- 加好友请求文案要按主题和线索来源生成，但不能夸大承诺。

建议新增状态：

| 状态 | 说明 |
| --- | --- |
| `queued` | 待自动添加 |
| `sent` | 已发送加好友 |
| `accepted` | 已通过 |
| `rejected` | 被拒绝 |
| `no_response` | 超期无响应 |
| `risk_paused` | 因风控暂停 |
| `failed` | 发送失败 |

当前 Core 里已有 `friend_requests` 暂存和审批模型，但还没有真实自动执行器。后续需要新增 `friend_add_executor`，并加入日限额、冷却时间、失败退避和账号风控。

## 5. 加微信后默认 C 类

C 类目标不是成交，而是自然补档。

C 类若水职责：

- 确认对方身份。
- 补公司、角色、品种、地区、资源链。
- 判断是否真和业务有关。
- 不谈敏感承诺。
- 不主动透露内部客户、人名、价格、合同、付款等信息。

C 类结果：

- 有业务关联且资料基本完整：申请升 B。
- 无直接业务关联：降 D，保持静默。
- 暂时不清楚：继续 C，低频沟通。

当前已实现的相关能力：

- C 级客户档案缺口识别。
- 内部补档任务。
- 补档任务 dry-run 预览。
- 补档任务确认后写 profile_patch。
- 满足条件后生成 pending C->B 升级申请。

待补：

- C->D 降级申请和静默规则。
- 自动加微信通过后自动建立 C 类客户档案。
- C 类低频沟通节奏控制。

## 6. B 类深入对接

B 类不是完全自动聊业务，而是可以做非敏感业务对接。

B 类若水职责：

- 继续补性格、喜好、沟通风格。
- 补成交前需要的深度资料。
- 判断真实需求、资源和成交意愿。
- 做公开资料解释、行业知识问答、流程说明。
- 识别供需、价格意向、付款方式、港口、货源等线索。
- 针对 B 类客户继续外部搜索，补完整背景调查。
- 涉及内部敏感信息时转人工。

B 类结果：

- 有真实成交意愿、资源清晰、沟通稳定，并带明确意向成交单：申请升 A。
- 发现不匹配：人工决定保留 B 或降 C/D。

待补：

- B 类背景调查任务。
- B->A 申请标准。
- B 类敏感信息识别和转人工队列。
- B 类成交意向单结构。

## 7. A 类人工介入

A 类全部由人工交流和对接。

若水职责：

- 客户总结。
- 关系分析。
- 跟进提醒。
- 沟通建议。
- 风险提示。
- 资料整理。
- 下一步动作建议。

A 类边界：

- 若水不自由承诺价格。
- 若水不自由承诺合同。
- 若水不自由承诺付款。
- 若水不自由承诺交期。
- 若水不自由承诺资源归属。
- 若水只做辅助、整理、提醒、建议和风险提示。

## 8. 建议的数据库分层

```text
lead_topics
  阶段性搜索主题、品类、关键词、范围、权重

lead_pre_pool
  全网搜索到的脏线索、模糊线索、待补资料线索

lead_add_queue
  审核程序判断可自动加微信的高相关线索

friend_add_attempts
  自动加微信执行记录、限额、结果、风控状态

customer_profiles / customer_authorizations
  加微信通过后的正式客户档案和分级授权

relationship_agent_events
  补档、观察、升级、降级、风控、跟进事件

customer_upgrade_requests
  C->B、B->A、A->S 等升级申请

customer_downgrade_requests
  C->D、B->C/D 等降级或静默申请，后续新增
```

## 9. 需要完善的规则

### 9.1 Lead 入预备库规则

需要定义：

- 哪些平台可信度高。
- 哪些关键词必须出现。
- 哪些字段至少满足一个。
- 电话/微信号怎么识别。
- 重复线索怎么合并。
- 风险词如何排除。

### 9.2 预备库 -> 待添加库规则

建议初始评分：

```text
总分 = 业务相关度 + 联系方式质量 + 主体可信度 + 品种匹配 + 地区价值 + 角色价值 - 风险分
```

待添加条件：

- 总分超过阈值。
- 有电话或微信号。
- 主营品种命中当前主题。
- 角色不是明显无关。
- 风险分低于阈值。

### 9.3 加微信限额规则

需要按账号维护：

- 每日上限。
- 每小时上限。
- 连续失败冷却。
- 被拒绝率阈值。
- 账号异常暂停。
- 不同来源的加好友文案模板。

### 9.4 C->B 规则

已基本实现，但还要补：

- 身份可信度。
- 资源链清晰度。
- 是否真实业务关联。
- C->D 降级触发。

### 9.5 B->A 规则

需要新增：

- 明确成交意向单。
- 真实供需或资源。
- 沟通稳定。
- 资料配合度。
- 风险可控。
- 人工审核通过。

## 10. 开发难点

1. 全网搜索数据很脏。
   - 同名公司、旧电话、无效微信、广告页、转载页会很多。
   - 当前不需要过度纠结，先靠规则优化和实践反馈自然完善。
   - 关键是保留来源、证据、时间和审核结果，方便后续调整规则。

2. 实体合并难。
   - 一个公司可能多个简称，一个人可能多个平台账号。
   - 当前以微信号为主键，所有线索围绕微信号展开。
   - 同一公司/人重复出现不作为重大问题，只要避免重复审核、重复加微信。

3. 自动加微信有账号风控。
   - 必须做日限额、退避、失败率监控和暂停机制。
   - 节奏比极限数量更重要：加人频率、入库节奏、预备库补充节奏要均匀。
   - 待添加库和预备库保持动态平衡，不多不少，刚刚好。
   - 预备库必须保留历史记录，避免同一线索重复审核、重复添加。

4. 审核程序需要持续校准。
   - 初期不要追求一次准确，应该保存误判样本，不断调整规则。
   - 审核标准可以按品类、主题和等级拆开，逐步形成多套可选规则。

5. C 类自然补档难。
   - 不能像填表一样审问客户，要结合对话时机，一次最多问一个问题。
   - 这部分主要交给专门的前期客户开发聊天 Agent 解决。
   - 后续应基于真实聊天记录训练/优化话术，让补档像正常业务交流。

6. B 类容易越过敏感边界。
   - 价格、合同、付款、客户名单、资源归属必须保持人工确认。
   - B 类重点不是承诺，而是补全信息、问问题、给线索，引导对方补全成交意向。
   - 若出现明确成交要素，再进入 B->A 申请。

7. A 类要彻底转人工。
   - 若水在 A 类只做辅助，不做业务承诺。

8. 自动化闭环要可审计。
   - 搜索、入预备库、进待添加库、自动加微信、转 C/B/A/D 都必须有证据和事件记录。

## 11. 开发建议顺序

P0:

1. 建 `lead_topics` 和 `lead_pre_pool`。
2. 建 Lead 初筛评分器。
3. 建预备库查询、导出和复核接口。

P1:

1. 建 `lead_add_queue`。
2. 建预备库 -> 待添加库审核程序。
3. 建加微信限额策略表和 dry-run。

P2:

1. 建自动加微信执行器。
2. 加好友通过后自动创建 C 类客户档案。
3. 接入现有 C 级补档任务和 C->B 升级申请。

P3:

1. 建 B 类背景调查任务。
2. 建 B->A 升级申请。
3. 建 A 类人工跟进工作台摘要。

## 12. 用户修正后的执行口径

### 12.1 数据脏不是当前核心阻塞

全网线索天然脏、重复、过期、真假混杂。第一阶段不追求清洗完美，先让系统跑起来。

做法：

- 先入预备库。
- 保存证据和来源。
- 保存审核结果。
- 保存被丢弃或保留的原因。
- 后续靠规则优化、误判样本和实际效果逐步校准。

### 12.2 微信号是客户开发主键

主体合并暂时不作为核心难点。

做法：

- 以微信号为主键。
- 电话、公司名、姓名、平台账号都作为微信号的辅助线索。
- 同一公司或同一人重复出现可以接受。
- 系统重点防止重复审核、重复加微信、重复沟通。

### 12.3 自动加微信强调节奏均匀

自动加微信不是越多越好，而是均匀、稳定、可持续。

做法：

- 每天限定人数。
- 每小时限定节奏。
- 根据待添加库容量，同步补充预备库。
- 预备库、待添加库、自动加微信队列保持动态平衡。
- 被拒绝、失败、无响应、异常都要记录，作为后续规则校准依据。

### 12.4 审核程序优先于人工逐条审批

从预备库进入待添加库，由程序审核。

人工职责：

- 设定阶段主题。
- 设定品类范围。
- 调整审核标准。
- 批量复核预备库。
- 修正规则权重。

### 12.5 C 类补档由专门聊天 Agent 执行

C 类不应像填表一样问客户。补档要通过自然聊天完成。

做法：

- 一次最多问一个问题。
- 业务问题优先回答。
- 根据对方话题顺势补公司、角色、品种、地区、资源链。
- 后续基于真实聊天记录训练/优化前期客户开发 Agent。

### 12.6 B 类职责是推进意向，不是做承诺

B 类重点：

- 补全客户背景。
- 问关键问题。
- 给公开资料和普通业务线索。
- 引导对方补全成交意向。
- 识别供需、价格意向、付款方式、港口、货源。

禁止：

- 承诺价格。
- 承诺合同。
- 承诺付款。
- 泄露内部客户、人名、公司名、资源归属。

### 12.7 A 类人工主导

A 类全部由人工交流和对接。

若水职责：

- 总结。
- 提醒。
- 分析。
- 风险提示。
- 资料整理。
- 跟进建议。

## 当前结论

用户修正后的方向应作为新主流程：

```text
全网线索 -> Lead 预备库 -> 程序审核 -> 待添加库 -> 限量自动加微信 -> C 类补档 -> B 类对接 -> A 类人工介入
```

核心变化：

- 人工不逐条审批加微信。
- 人工主要设定主题、修正规则、复核批量样本。
- 自动加微信可以做，但必须有日限额和风控暂停。
- A 类必须由人工主导。
