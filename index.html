最终目标架构：一个业务系统，四个清晰平面

业务生命周期平面

  • Campaign → Evidence → Lead
  • Qualification → Engagement
  • Conversation → C/B/A → Handoff
  • Project A 是业务应用，不是第二个 Core

控制平面：Ruoshui Core

  • PolicyContext 统一规则输入
  • Workflow 状态、暂停、恢复、重试
  • Action Gateway 唯一外部动作出口
  • 授权、审计、幂等和人工接管

能力与数据平面

  • Provider Registry：采集/解析/模型/执行
  • Unified SQLite：唯一事实库
  • Evidence lineage：来源→处理→候选
  • 知识、客户记忆和审计分域不分库

体验与运维平面

  • 微信 Middleware 单一通道
  • Chatwoot 仅做人工作台
  • Prometheus 指标，逐步兼容 OTel
  • 总图、告警、审核和接管界面

旧板块最终归属继续保留只做的改造禁止重复建设

成熟经验后的技术决策：现在采用、只借模式、明确延后

能力结论当前主选成熟经验允许升级的客观条件
静态/动态采集采用Scrapy + Playwright成熟抓取、并发、重试与浏览器自动化；保持隔离 Worker站点规模超过单机吞吐且浏览器池成为瓶颈
正文与身份采用Trafilatura + phonenumbers + RapidFuzz结构化工具优先，LLM 只处理歧义离线基准连续低于目标准确率才替换
长工作流借模式，暂不部署Core DB 状态机 + APScheduler借鉴 Temporal 的 durable execution、幂等、暂停/恢复跨天流程量、补偿复杂度或故障恢复超过现有状态机能力
策略引擎借模式，暂不部署Core PolicyContext借鉴 OPA 的策略/执行分离和可测试决策多服务共享策略、规则发布频繁且需要独立策略团队
遥测渐进采用Prometheus，兼容 OpenTelemetry 语义指标、日志、追踪统一关联；业务审计仍留 DB跨进程排障需求稳定出现时接 OTel Collector
数据血缘采用模式Evidence contract lineage借鉴 OpenLineage 的来源、运行、产物可追溯暂不部署独立 lineage 服务，先固化字段
人工工作台旁路采用Chatwoot共享 inbox、人工协作和接管，不做业务大脑API token、身份映射和只读/接管边界验收后扩大使用
Agent 框架不进入主链现有 Core + ProviderLangGraph/AutoGen 可参考编排模式，但会形成第二状态机只有决定替换 Core 且完成全量迁移论证时再评估
向量库/图数据库暂不采用SQLite / FTS5 / 关系表先用真实查询性能证明需求，不以“架构先进”为理由引入P95 查询、数据规模或关系遍历出现可复现瓶颈

防止反复开发后弃用：五道投资闸门

1. 问题证据必须有失败率、延迟、准确率、运维成本或业务损失数据。
2. 所有权唯一新模块必须替代或补足一个明确能力，不能复制 Core 权威。
3. 隔离试验先 Provider/旁路/临时库，定义输入输出和退出开关。
4. 验收基准上线前有离线集、回归、故障注入和回滚标准。
5. 晋级或删除达到指标才进入主链；连续两轮无收益则删除试验层。

Public Search / ScrapyPlaywright 动态渲染
  • 域名白名单与 SSRF 防护
  • 只做公开 GET 与原始证据输出
  • 无数据库、Core、微信权限

3. 清洗、身份与候选

Trafilatura 正文phonenumbers / RapidFuzz
  • 自动电话发现与 E.164 标准化
  • 模糊身份匹配只作辅助
  • 输出 Evidence / Candidate JSON
→

4. Ruoshui Core 唯一裁决

Policy / Memory / AuditProvider Resolution / Action Gateway
  • 显式审核后才可写 Lead 表
  • 真实加微信与发送必须走 Core
  • C / B / A 等级与授权唯一归口
→

5. 执行与经营闭环

WeChat Middleware
  • 加好友、结果回收、C 类聊天
  • B 类非敏感互动
  • A 类人工主导
  • 所有动作留痕与可停止

不可跨越的权限边界

Core 是唯一最终裁决外部库、模型与 Worker 不拥有客户状态、授权、记忆或发送权。
Unified SQLite 是唯一事实库不引入第二套工作流状态库，不迁移权威客户数据。
Provider 只产生证据公开采集、正文抽取与身份辅助均输出候选，由 Core 显式接收。
微信保持单一通道仅使用现有 Middleware，不并行引入 Wechaty 或其他协议栈。
